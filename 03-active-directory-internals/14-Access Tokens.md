# Access Tokens (Security Perspective)

## Overview

An **Access Token** is a kernel-managed security object created after a successful authentication. It represents the authenticated identity of a user or process and contains all the information Windows needs to make authorization decisions.

From a security perspective:

> Access Tokens are the foundation of Windows authorization. Every access check performed by the operating system relies on the information stored inside an Access Token.

Understanding Access Tokens is essential for:

- Windows Internals
- Active Directory Security
- Authorization
- Privilege Escalation
- Token Impersonation
- Incident Response
- Red Team Operations

---

# What is an Access Token?

An Access Token is generated by **LSASS** after successful authentication.

It contains:

- User SID
- Group SIDs
- Privileges
- User Rights
- Integrity Level
- Logon Session Information
- Default DACL

Every process started by the user inherits an Access Token.

# User SID

Every Access Token contains the authenticated user's SID.

Example:

```
S-1-5-21-...
```

This uniquely identifies the user.

---

# Group SIDs

The token also contains every group the user belongs to.

Example:

```

в†“

IT Department

в†“

```

Windows evaluates all group memberships during authorization.

---

# Privileges

Privileges define special system capabilities.

Examples:

| Privilege | Purpose |
|-----------|----------|
| SeDebugPrivilege | Debug other processes |
| SeBackupPrivilege | Read any file for backup |
| SeRestorePrivilege | Restore files |
| SeShutdownPrivilege | Shutdown system |
| SeImpersonatePrivilege | Impersonate another user |
| SeTakeOwnershipPrivilege | Take ownership of objects |

Privileges differ from permissions.

---

# Integrity Level

Access Tokens include a Mandatory Integrity Level (MIL).

Common levels:

| Integrity Level | Description |
|-----------------|-------------|
| Low | Sandboxed applications |
| Medium | Standard user |
| High | Administrator |
| System | Operating System |

Integrity Levels enforce Mandatory Integrity Control (MIC).

---

# Logon SID

Each interactive logon session receives a unique:

```
Logon SID
```

It identifies the current login session rather than the user account.

---

# Default DACL

The token includes a Default Discretionary Access Control List.

Used when:

- Creating files
- Creating registry keys
- Creating processes
- Creating kernel objects

The Default DACL determines initial permissions.

---

# Authentication ID (LUID)

Each Access Token contains a:

```
Locally Unique Identifier (LUID)
```

Used internally to identify the logon session.

---

# Security Importance

Access Tokens determine:

- User identity
- Administrative privileges
- Group memberships
- Integrity level
- Resource permissions

Without an Access Token, a process cannot access protected resources.

---

# Offensive Perspective

Attackers target Access Tokens because they represent authenticated identities.

Common objectives include:

- Token theft
- Token duplication
- Token impersonation
- Privilege escalation
- Lateral movement

Rather than stealing passwords, attackers may attempt to steal or abuse tokens.

---

# Common Token Attacks

| Attack | Description |
|----------|-------------|
| Token Impersonation | Use another user's token |
| Token Duplication | Copy an existing token |
| Token Theft | Steal privileged token |
| Pass-the-Token | Reuse stolen token |
| Potato Attacks | Abuse SeImpersonatePrivilege |

Most token attacks abuse legitimate Windows functionality.

---

# High-Value Tokens

Attackers seek tokens belonging to:

- SYSTEM
- Domain Admins
- Enterprise Admins
- Local Administrators
- Service Accounts

These tokens often provide elevated privileges.

---

# Defensive Perspective

Administrators should:

- Limit administrative logons
- Protect LSASS
- Restrict SeImpersonatePrivilege
- Enable Credential Guard
- Monitor token-related events
- Remove unnecessary privileges
- Apply least privilege

Token security is critical for preventing privilege escalation.

---

# Access Token vs Credentials

| Access Token | Credentials |
|--------------|-------------|
| Created after authentication | Used for authentication |
| Represents identity | Verifies identity |
| Used for authorization | Used during logon |
| Exists in memory | Passwords, hashes, smart cards |

Credentials create the token; the token is then used for all subsequent access checks.

# Why Access Tokens Matter

Understanding Access Tokens enables security professionals to:

- Analyze Windows authorization
- Investigate privilege escalation

- Secure Windows environments
- Assess enterprise authentication

---

# Key Takeaways

- Access Tokens are created after successful authentication.
- They represent the security context of a user or process.
- Tokens contain SIDs, group memberships, privileges, and integrity levels.
- Windows authorizes access by comparing Access Tokens against ACLs.
- Primary Tokens belong to processes, while Impersonation Tokens belong to threads.
- Token theft and impersonation are common techniques used in privilege escalation attacks.

---

# Key Insight

> Authentication ends when Windows creates an Access Token. From that point forward, every authorization decision depends on the information inside the token not on the user's password. Understanding Access Tokens is essential because they are the operating system's trusted representation of identity and privilege.
