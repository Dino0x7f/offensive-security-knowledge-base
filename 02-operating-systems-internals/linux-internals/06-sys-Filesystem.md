# sys Filesystem (Security Perspective)

## Overview

The **/sys** filesystem (sysfs) is a virtual filesystem that exposes information about the Linux kernel, hardware devices, drivers, and kernel subsystems.

Unlike traditional filesystems, `/sys` does **not** store data on disk. Its contents are dynamically generated by the kernel and provide a structured interface for inspecting and configuring kernel objects.

From a security perspective:

> The `/sys` filesystem exposes how the Linux kernel sees the system. It is a critical source of information for hardware enumeration, kernel configuration, device management, and security monitoring.

Understanding `/sys` is essential for Linux internals, kernel security, driver analysis, incident response, and system hardening.

---

# What is sysfs?

`sysfs` is a virtual filesystem mounted at:

```
/sys
```

It exposes kernel objects such as:

- Hardware devices
- Device drivers
- Kernel modules
- Block devices
- Network interfaces
- CPU information
- Memory information
- Kernel parameters

Unlike `/proc`, which focuses on processes and runtime state, `/sys` represents the kernel's internal object model.

```
Applications

↓

Read /sys
```

---

# Why /sys Exists

The `/sys` filesystem provides a standardized interface for:

- Hardware discovery
- Device management
- Driver interaction
- Kernel configuration
- System monitoring

Many administration tools rely on `/sys` rather than parsing kernel structures directly.

---

# Main Directories

## /sys/class

Contains logical device classes.

Examples:

```
/sys/class/net/

/sys/class/block/

/sys/class/power_supply/
```

Security relevance:

Useful for enumerating available devices.

---

## /sys/devices

Represents the physical device tree.

Contains:

- CPUs
- PCI devices
- USB devices
- Storage controllers
- Network adapters

Security relevance:

Provides hardware fingerprinting information.

---

## /sys/block

Lists block storage devices.

Examples:

```
sda

sdb

nvme0n1
```

Useful for identifying storage devices and partitions.

---

## /sys/module

Contains loaded kernel modules.

Example:

```
/sys/module/
```

Security relevance:

Allows investigators to determine which kernel modules are currently loaded.

---

## /sys/kernel

Contains kernel-level information.

Examples:

```
/sys/kernel/debug

/sys/kernel/security
```

Some files require elevated privileges.

---

## /sys/fs

Contains filesystem-specific kernel information.

Examples:

```
cgroup

bpf

ext4
```

Security relevance:

Provides information about modern kernel features such as eBPF and cgroups.

---

# Hardware Information

Examples include:

CPU:

```
/sys/devices/system/cpu/
```

Memory:

```
/sys/devices/system/memory/
```

NUMA:

```
/sys/devices/system/node/
```

These directories expose hardware topology and configuration.

---

# Network Interfaces

```
/sys/class/net/
```

Contains one directory for each interface.

Examples:

```
eth0

wlan0

lo
```

Information includes:

- MAC address
- MTU
- Interface state
- Speed
- Statistics

Security relevance:

Useful during network enumeration.

---

# Kernel Modules

```
/sys/module/
```

Each loaded kernel module has its own directory.

Information includes:

- Module name
- Parameters
- Dependencies
- Reference count

Security relevance:

Attackers and defenders inspect loaded modules for:

- Rootkits
- Malicious drivers
- Unexpected modules

---

# Device Drivers

Driver information is available through:

```
/sys/bus/

/sys/devices/

/sys/class/
```

This allows administrators to determine:

- Which driver controls a device
- Device-driver relationships
- Driver configuration

---

# Kernel Parameters

Some kernel parameters are exposed through `/sys`.

Examples:

```
/sys/kernel/

/sys/devices/system/
```

Unlike `/proc/sys`, many `/sys` entries relate directly to hardware and kernel objects.

Some values can be modified by privileged users.

---

# Relationship with the Kernel

```
User Process

↓

Read /sys

↓

Kernel Objects

↓

Live System Information
```

The kernel generates all information dynamically.

---

# /proc vs /sys

| /proc | /sys |
|--------|-------|
| Process information | Kernel objects |
| Runtime state | Device model |
| Process-centric | Hardware-centric |
| Process directories | Device directories |
| System statistics | Driver & hardware information |

---

# Security Importance

The `/sys` filesystem provides valuable information about:

- Hardware devices
- Drivers
- Kernel modules
- CPU configuration
- Network interfaces
- Storage devices

This information assists in both system administration and security analysis.

---

# Common Security Risks

Improper access to `/sys` may allow:

- Information disclosure
- Hardware fingerprinting
- Kernel configuration changes
- Device manipulation (if writable)

Most sensitive entries require root privileges.

---

# Offensive Perspective

Attackers inspect `/sys` to:

- Identify hardware
- Discover network interfaces
- Enumerate storage devices
- Detect virtualization
- Detect containers
- Enumerate loaded kernel modules
- Understand the execution environment

This information helps tailor exploitation techniques.

---

# Defensive Perspective

Security teams use `/sys` to:

- Verify loaded kernel modules
- Audit hardware configuration
- Detect unauthorized devices
- Monitor driver state
- Investigate system integrity
- Validate kernel configuration

---

# Key Takeaways

- `/sys` is a virtual filesystem representing the Linux kernel's object model.
- It provides information about hardware, drivers, kernel modules, and devices.
- Unlike `/proc`, which focuses on processes, `/sys` focuses on kernel-managed objects.
- Many administrative and monitoring tools rely on `/sys` for hardware and kernel information.
- Both attackers and defenders use `/sys` extensively during system enumeration and security analysis.

---

# Key Insight

> The `/sys` filesystem is the Linux kernel's window into the hardware and device layer. It exposes how the kernel organizes, manages, and interacts with system components, making it an essential resource for understanding both system behavior and potential attack surfaces.
